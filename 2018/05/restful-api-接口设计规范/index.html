Restful API 接口设计规范

2018-05-07

文章目录

基础

API版本符合语义化版本控制

版本格式：主版本号.次版本号.修订号，版本号递增规则如下：

  1. 主版本号：当你做了不兼容的 API 修改，
  2. 次版本号：当你做了向下兼容的功能性新增，
  3. 修订号：当你做了向下兼容的 bug 修复。

其他标签，如预发版本、构建元信息，可以拓展到“主版本号.次版本号.修订号”的后面。

强制在路径中使用主版本号

制定版本并在版本之间平缓过渡对于设计和维护一套API是个巨大的挑战。所以，最好在设计之初就使用一些方法来预防可能会遇到的问题。

为了避免API的变动导致用户使用中产生意外结果或调用失败，最好强制要求所有访问都需要指定版本号。请避免提供默认版本号，一旦提供，日后想要修改它会相当困难。

虽然可将版本号信息放置在头信息(HTTP Headers)中，譬如在Accept段中标明版本信息。例如:

Accept: application/json; version=3

但是在路径中标识版本信息将更加直观；并且若系统同时提供多个版本的服务，各个版本之间将不受影响。例如：

get /v1/cmdb/clusters

get /v2/cmdb/clusters

新老版本可以共同存在，新版本将慢慢替换老版本。

使用自定义 Header指明api的具体版本

虽然要求同一主版本的api需要向下兼容，但是指明具体的api版本能方便消费者调试与查阅文档

 X-Api-Version: v1.2.0
URI失效

随着系统发展，总有一些API失效或者迁移，对失效的API，返回404 not found 或 410 gone；对迁移的API，返回 301 重定向。

为内省而提供 Request-Id

为每一个请求响应包含一个Request-Id头，并使用UUID作为该值。通过在客户端、服务器或任何支持服务上记录该值，它能为我们提供一种机制来跟踪、诊断和调试请求。

提供人类可读的文档

提供人类可读的文档让客户端开发人员可以理解你的API。使用swagger生成人类可读的api文档。

请求（Requests）

资源名为复数形式

使用复数形式为资源命名，除非这个资源在系统中是单例的 (例如，在大多数系统中，给定的用户帐户只有一个)。 这种方式保持了特定资源的统一性。

例如：

/openstack-api/v1/servers
路径要小写

为了和域名命名规则保持一致，使用小写字母并用-分割路径名字，例如：

/openstack-api/v1/servers
资源集合与单个资源

资源集合：

/openstack-api/v1/regions 
/openstack-api/v1/regions/1/servers

单个资源：

/openstack-api/v1/servers/1 
/openstack-api/v1/servers/1,2,3
最小化路径嵌套

在一些有父路径/子路径嵌套关系的资源数据模块中，路径可能有非常深的嵌套关系，例如:

/regions/{region_id}/connections/{connection_id}/servers/{server_id}

推荐在根(root)路径下指定资源来限制路径的嵌套深度。使用嵌套指定范围的资源。在上述例子中，servers属于connections，connections属于regions可以表示为：

/regions/{region_id}
/regions/{region_id}/connections
/connections/{connection_id}
/connections/{connection_id}/servers
/servers/{server_id}
通过标准HTTP方法对资源CRUD

GET: 查询

GET /servers
GET /servers/1
GET /servers/1/volumns

POST: 创建单个资源。POST一般向“资源集合”型URI发起；

POST /servers 
POST /regions/1/servers

PUT：更新单个资源（全量），客户端提供完整的更新后的资源。与之对应的是 PATCH，PATCH 负责部分更新，客户端提供要更新的那些字段。PUT/PATCH一般向“单个资源”型uri发起

PUT /servers/1
PUT /regions/1

DELETE：删除

DELETE /regions/1/servers/2
DELETE /servers/1,2,3

ACTIONS: 指定行为

特殊情况下，需要为某些资源指定行为。为了描述清楚，在行为前加上一个标准的actions：

/resources/{resource_id}/actions/:action

例如：

/servers/1/actions/stop   //停止id为1的云主机
安全性与幂等性

安全性：不会改变资源状态，可以理解为只读的； 幂等性：执行1次和执行N次，对资源状态改变的效果是等价的。

安全性 幂等性
GET 是 是
POST 否 否
PUT 否 是
DELETE 否 是

安全性和幂等性均不保证反复请求能拿到相同的response。以 DELETE 为例，第一次DELETE返回200表示删除成功，第二次返回404提示资源不存在，这是允许的。

复杂查询

查询可以捎带以下参数：

示例 备注
过滤条件 ?type=1&age=16 允许一定的ur冗余，如/zoos/1 与 /zoos?id=1
排序 ?ordering=age 使用ordering标识需要排序的字段
分页 ?page=1&page_size=10 page 为页数，page_size 为页面大小
搜索 ?search=abc search标识需要搜索的关键字
在请求的body体使用JSON格式数据

在PUT/PATCH/POST请求的正文（request bodies）中使用JSON格式数据，而不是使用 form 表单形式的数据。这与我们使用JSON格式返回请求相对应，例如:

$ curl -X POST https://service.com/apps \
    -H "Content-Type: application/json" \
    -d '{"name": "demoapp"}'

{
  "id": "01234567-89ab-cdef-0123-456789abcdef",
  "name": "demoapp",
  "owner": {
    "email": "username@example.com",
    "id": "01234567-89ab-cdef-0123-456789abcdef"
  },
  ...
}

响应(Response)

返回合适的状态码

为每一次的响应返回合适的HTTP状态码。 好的响应应该使用如下的状态码:

状态码 状态 方法 说明
200 OK GET 客户端向服务器请求数据，服务器成功找到它们
201 CREATED POST/PUT 客户端向服务器提供数据，服务器根据要求创建了一个资源
204 NO CONTENT DELETE 客户端要求服务器删除一个资源，服务器删除成功
400 INVALID REQUEST POST/PUT 客户端向服务器提供了不正确的数据，服务器什么也没做
401 Unauthorized * 客户端未通过HTTP认证。（token失效，或者用户提供的认证信息有误）
403 Forbidden * 客户端访问了未授权的资源，权限未通过。
404 NOT FOUND * 客户端引用了一个不存在的资源或集合，服务器什么也没做
*406 Not Acceptable GET 用户请求的格式不可得（比如用户请求JSON格式，但是只有XML格式）
*410 Gone GET 用户请求的资源被永久删除，且不会再得到的。
*422 Unprocesable entity POST/PUT 当创建一个对象时，发生一个验证错误。
500 INTERNAL SERVER ERROR * 服务器发生内部错误，客户端无法得知结果，即便请求已经处理成功
*503 Internal Server Error * 服务器暂时处于超负荷或正在进行停机维护，现在无法处理请求。
提供全部可用的资源

提供全部可显现的资源表述 (例如： 这个对象的所有属性) ，当响应码为200或是201时返回所有可用资源，包含PUT/PATCH和DELETE请求，例如:

{
    "id": 0,
    "created_time": "2018-05-08T02:21:03Z",
    "region_id": "string",
    "region_name": "string",
    "updated_time": "2018-05-08T02:21:03Z"
}
分页响应格式
{
  	"pagination": {
    	"page": 0,
    	"page_size": 0,
    	"total": 0
  	},
 	"results": [
 		{...},
 		{...}
 		]
}
提供标准的时间戳

为资源提供默认的创建时间created_time和更新时间updated_time，例如:

{
    "id": 0,
    "region_id": "string",
    "region_name": "string",
    "created_time": "2018-05-08T02:21:03Z",
    "updated_time": "2018-05-08T02:21:03Z"
}
使用UTC时间

仅接受和返回UTC格式的时间。ISO8601格式的数据，例如:

{
    "created_time": "2018-05-08T02:21:03Z"
}
嵌套外键关系

使用嵌套对象序列化外键关联，例如:

{
  "name": "service-production",
  "owner": {
    "id": "5d8201b0..."
  },
  // ...
}
错误处理

  • 不要发生了错误但给2xx响应，客户端可能会缓存成功的http请求；

  • 正确设置http状态码，不要自定义；

  • Response body 使用统一的错误格式

    {
"code": "string",
"message": "string",
"errors": []
}

    code 使用类型的编码，不要使用数字类型，用于错误定位与前端判断逻辑。

    message 用于提示，简短的错误描述，用于页面提示

    errors 存储详细的错误信息，例如字段格式错误等。

业务异常

由自己的业务代码抛出，表示一个用例的前置条件不满足、业务规则冲突等，比如参数校验不通过、权限校验失败。

业务类异常必须提供2种信息：

  1. 如果抛出该类异常，HTTP 响应状态码应该设成什么；
  2. 异常的文本描述；

非业务异常

表示不在预期内的问题，通常由类库、框架抛出，或由于自己的代码逻辑错误导致，比如数据库连接失败、空指针异常、除0错误等等

非业务类异常提供2种信息：

  1. HTTP响应状态码统一设置为500
  2. 线上可以统一文案如“服务器端错误，请稍后再试”
